You are upgrading the Torque Assist portion of TorqueShed.pro into a polished, trust-building automotive diagnostics helper for DIY mechanics, techs, and enthusiasts.

The current Torque Assist experience feels cheap because it repeats the same information, gives generic “follow these steps” answers, and does not behave like a real diagnostic assistant with memory, narrowing logic, or confidence-based reasoning.

Your task is to redesign and rebuild Torque Assist into a professional-grade diagnostics workflow that feels like a real shop foreman helping the user isolate faults.

PRIMARY OBJECTIVE
Build Torque Assist as an interactive diagnostic interview and reasoning engine, not a generic chatbot.
It must:
1. collect structured symptom data
2. remember what has already been asked and answered
3. avoid repeating the same guidance
4. narrow possible causes over time
5. produce prioritized likely causes with reasoning
6. generate a staged test plan
7. update recommendations based on test results
8. feel professional, clean, and credible

CORE PRODUCT RULES
- Never repeat the same advice block unless new evidence changes priority
- Never restate the user’s symptoms over and over in slightly different wording
- Never dump a long wall of generic steps
- Ask only the most useful next question when uncertainty is high
- Prefer narrowing questions over broad open-ended questions
- Track all previously collected answers in session memory
- Show the user progress through diagnosis
- Present confidence levels for likely causes
- Always explain WHY a question is being asked or WHY a test matters
- Keep tone professional, practical, and mechanic-friendly
- Make the assistant feel like a diagnostic helper, not a content generator

PHASE 3: SMART NARROWING
Based on category, dynamically ask the next best question.
Examples:
- no crank: single click, rapid click, silence, dash lights dim, starter relay behavior, jump start effect
- overheating: highway or idle only, coolant loss, fan operation, heater behavior, pressure buildup, white smoke
- misfire: cold/hot, single cylinder or random, under load or idle, fuel smell, recent tune-up, code pattern
- charging issue: battery voltage off/running, load behavior, belt condition, warning lights, alternator replacement history
- suspension noise: speed dependent, brake-dependent, turning-dependent, one side or both, over bumps only, steering angle effect

PHASE 4: HYPOTHESIS ENGINE
Maintain a structured list of possible causes with:
- cause name
- confidence score
- supporting evidence
- contradictory evidence
- recommended next confirming test
- estimated difficulty
- estimated cost range
- safety level

As answers come in, update and rerank likely causes rather than resetting or repeating.

PHASE 5: TEST PLAN
Output a staged test plan:
- Step 1: easiest / safest / highest-value check
- Step 2: next best discriminator
- Step 3: confirmatory test
- Step 4: repair direction if confirmed

Each step should include:
- purpose of the test
- tools needed
- expected good result
- expected bad result
- what to do next based on the result

PHASE 6: RESULTS LOOP
After each user-entered test result, update the hypotheses and next actions.
Do not repeat previous steps that are already completed.
Mark completed tests clearly.
Persist findings in session state.

ANTI-REPETITION LOGIC
Implement strict anti-repetition safeguards:
- hash or fingerprint prior assistant advice blocks
- compare proposed new response against prior response blocks

RESPONSE FORMAT
Every assistant response should follow this format when applicable:

1. Current read
A concise one-paragraph assessment of where the diagnosis stands right now.

2. Most likely causes
A ranked list with confidence percentages.

3. Best next question or best next test
Only one primary next action unless there is a justified branch split.

4. Why this matters
Short explanation of how the answer/test narrows the fault.

5. What changes based on the result
Show the branch logic.

Example:
Current read:
Based on your no-crank condition, full dash power, and single heavy click from the starter area, the problem is currently leaning toward a starter motor or high-resistance power/ground path rather than a dead battery.

Most likely causes:
- Failed starter motor or solenoid (62%)
- High resistance on battery cable or engine ground (24%)
- Starter relay / control issue (9%)
- Mechanical engine lockup (5%)

Best next test:
Measure battery voltage at rest, then at the starter main lug while attempting crank.

Why this matters:
This separates a control issue from a power delivery issue in one step.

What changes based on the result:
- If voltage stays strong at battery but drops badly at starter, suspect cable or connection resistance.
- If voltage reaches starter correctly and it only clicks, suspect starter failure.
- If no trigger signal reaches solenoid, move upstream to relay/ignition/control logic.
